# Spec Guardian — Acceptance Stage

本子 Skill 覆盖验收阶段：测试验收、归档、检查清单。

## 阶段范围

测试验收 → 归档关闭

## 规则（if-then）

```
IF 代码未开发完成 → 阻止验收，提示完成开发
IF 测试未全部通过 → 阻止验收，提示修复失败测试
IF 测试全部通过 AND 验收报告未创建 → 提示创建验收报告
IF 验收报告已创建 AND 未通过 → 阻止归档，提示按报告修订
IF 验收报告已通过 → 执行归档前检查清单
IF 归档前检查清单全部通过 → 允许归档
IF 归档 → 先归档任务 Spec，后归档阶段 Spec，最后归档设计文档和执行计划
```

## 评审准入/准出标准

### 准入

- 文档结构完整，必填字段无空缺。
- 文件头包含版本、对应上游文档、状态、评审记录占位。
- 作者已完成自检（按预审清单）。
- 所有上游依赖文档状态为「已通过」或「已完成，已归档」。

### 准出

**通过**：无 P0，无未解决 P1，P2 可遗留。
**不通过**：存在 P0，或存在未解决 P1。

### 重评规则

- 不通过需修订后重新提交。
- 最多 3 轮正式评审；超过则召开专题会议。
- P2 可现场修订，结论仍为「通过」。

## Spec 评审预审清单

### 完整性
- [ ] 目标、范围、交付物清单清晰。
- [ ] 实现步骤可执行、无重大遗漏。
- [ ] 测试用例覆盖正常/异常/回归路径。
- [ ] 验收标准可逐项核查。

### 一致性
- [ ] 技术方案与上游设计文档一致。
- [ ] 任务拆分与执行计划一致。
- [ ] 新增类型/接口/参数在所有模块同步。

### 可落地性
- [ ] 代码片段可直接使用或与架构兼容。
- [ ] 测试依赖明确。
- [ ] 风险与应对措施具体。

## 归档前最终检查清单

- [ ] 代码已开发完成并合并。
- [ ] 所有测试已通过。
- [ ] 验收专家已出具通过结论。
- [ ] 用户文档已同步更新。
- [ ] 验收报告已创建。
- [ ] 类型检查、构建、lint 已通过。
- [ ] 所有 P0/P1 已闭环。
- [ ] 所有任务 Spec 状态已更新为「已完成，已归档」。
- [ ] 阶段 Spec 状态后更新为「已完成，已归档」。
- [ ] 设计文档、执行计划状态已同步更新。

## 阶段转换检查点（验收 → 归档）

| 检查项 | 通过条件 | 失败动作 |
|---|---|---|
| 验收报告 | 已创建且结论为通过 | 阻止，提示修订 |
| 归档前检查清单 | 全部通过 | 阻止，提示补充缺失项 |
