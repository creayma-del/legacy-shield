# Spec Guardian — Scenarios

本子 Skill 列举 spec-guardian 的常见场景处理示例及违规操作处理规则。

## 常见场景处理示例

### 场景 1：用户要求直接开始代码开发

**用户说**：「现在开始写代码实现功能 X。」

**处理步骤**：

1. 检查是否存在功能 X 对应的阶段 Spec 和所有任务 Spec。
2. 若阶段 Spec 不存在 → 阻止，提示需先完成完整流程。
3. 若阶段 Spec 状态非「已通过」→ 阻止，提示当前状态及缺失的评审环节。
4. 若存在未评审通过的任务 Spec → 阻止，提示等待任务 Spec 评审完成。
5. 若阶段 Spec 和所有任务 Spec 状态均为「已通过」→ 允许，提醒严格按 Spec 开发。

### 场景 2：用户要求修改已归档 Spec

**用户说**：「在 phase-v1.0-spec.md 里加一条规则。」

**处理步骤**：

1. 读取文件状态，确认其为「已完成，已归档」。
2. 阻止直接修改。
3. 询问改动性质：
   - 错别字/示例修正 → 走 PATCH 流程，在补丁 Spec 中说明。
   - 能力变更或新增 → 走新版本流程（如 v1.1 或 v2.0）。

### 场景 3：用户要求只改一行文档

**用户说**：「README 里有个错别字，我直接改一下。」

**处理步骤**：

1. 判断是否为 PATCH 级别修复。
2. 若是 → 允许走 PATCH 流程，创建或更新补丁 Spec，记录修改原因。
3. 若改动涉及用户可见的功能描述 → 可能需要 MINOR 流程，需进一步确认。

### 场景 4：用户坚持绕过流程

**用户说**：「没时间走流程了，先写代码。」

**处理步骤**：

1. 明确拒绝并说明纪律要求。
2. 提供最小可行路径：如果是 PATCH，可走简化流程；如果是 MINOR，至少先完成需求对齐和简化的需求分解。
3. 记录用户坚持绕过的意图，但不执行代码修改。
4. 如需升级，提示用户联系项目负责人特批。

### 场景 5：多任务 Spec 状态不一致

**情况**：阶段 Spec 已通过，但某个任务 Spec 仍在评审中。

**处理步骤**：

1. 以任务 Spec 状态为准，该任务不能进入代码开发。
2. 其他已通过任务 Spec 的任务可并行开发。
3. 提示用户等待未通过任务 Spec 的评审完成。

### 场景 6：用户要求跳过任务 Spec 直接开发

**用户说**：「阶段 Spec 已经通过了，我先写代码，任务 Spec 后面补。」

**处理步骤**：

1. 明确拒绝。
2. 说明规则：阶段 Spec 通过后，必须先生成并评审通过所有任务 Spec，才能进入代码开发。
3. 如用户认为任务无需拆分为独立 Spec，应先将该任务合并入阶段 Spec 或调整执行计划，再重新评审。

### 场景 7：设计文档/执行计划未评审就进入下一阶段

**用户说**：「设计文档写完了，直接开始写阶段 Spec 吧。」

**处理步骤**：

1. 读取设计文档和执行计划的状态。
2. 若任一文档状态非「已通过」→ 阻止，提示必须先完成评审并获批准。
3. 若用户询问是否可以边写 Spec 边评审设计文档 → 明确拒绝，阶段 Spec 必须在设计文档和执行计划评审通过后编写。

## 处理违规操作

当检测到以下情况时，必须阻止并引导用户回到正确流程：

1. **未对齐先设计**：提醒用户先召开需求对齐会议并产出双方签字的会议纪要。
2. **未分解先设计**：提醒用户先完成《需求分解文档》并获得项目负责人批准。
3. **未审先开发**：提醒用户 Spec 尚未通过评审，需先完成评审。
4. **擅自修改已归档文档**：提醒用户已归档文档不可直接修改，需通过新版本流程或 PATCH 流程重新发起。
5. **范围蔓延**：提醒用户新需求应单独走新版本流程，不得混入当前 Spec。
