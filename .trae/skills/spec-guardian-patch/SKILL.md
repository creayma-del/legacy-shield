# Spec Guardian — Patch

本子 Skill 定义 PATCH 流程、PATCH 与任务 Spec 的关系，以及 Spec 批准后的范围变更处理。

## PATCH 流程

适用于不影响架构、不修改公共接口、不引入新能力的 bug 修复、文档修正、性能微调。

### 判定条件

必须同时满足：

- 不新增 API / CLI 参数 / 配置项。
- 不修改现有接口签名或返回结构。
- 不引入新的外部依赖。
- 不影响已有测试的断言目标（可新增测试，但不可破坏已有测试）。

### 允许的简化

- 可在现有版本 Spec 中新增「PATCH 任务」小节，不必创建全新阶段 Spec。
- 需求对齐会议和需求分解文档可合并为简化的「补丁影响评估」。
- 设计文档如无需变更，可仅做引用说明。

### 仍需强制通过

- 补丁任务 Spec 必须通过评审。
- 如涉及设计变更，必须同步更新设计文档。
- 必须通过测试验收。

### PATCH 与任务 Spec 的关系

PATCH 不强制要求创建独立的任务 Spec 文件，但必须以某种形式通过 Spec 评审：

- **默认方式**：在现有阶段 Spec 中新增「PATCH 任务」小节，包含变更描述、影响范围、测试计划、验收标准。
- **可选方式**：若 PATCH 影响范围跨多个模块，或需要独立跟踪进度，可创建一个独立的 PATCH 任务 Spec 文件（命名如 `phase-v{x}.{y}-patch-{n}-spec.md`）。
- **禁止方式**：未形成任何 Spec 文档就直接修改代码或合并。

### 紧急安全修复

- 可先修复并合并，但事后必须在 24 小时内补全 Spec 与评审记录。
- 需记录紧急原因、影响范围、回滚方案。

## Spec 批准后范围变更处理

一旦阶段 Spec 状态变为「已通过」，其范围冻结。任何偏离都必须按以下规则处理：

### 小型偏差（可在 PATCH 中消化）

- 不影响 Spec 核心目标。
- 不修改已评审通过的技术方案。
- 不引入新的能力或接口。
- 处理：在 PATCH 任务中记录，走 PATCH 流程。

### 中型偏差（需升级 Spec）

- 影响某个验收标准或实现细节，但不改变整体范围。
- 处理：修订阶段 Spec，重新进入评审（状态从「已通过」回到「评审中」或创建补丁 Spec 并评审）。

### 大型偏差（需新版本）

- 改变核心目标、新增能力、修改接口或架构。
- 处理：走新版本流程（MINOR 或 MAJOR），原 Spec 保持已归档状态。

### 已开发代码的处理

- PATCH 消化：继续开发，补充记录。
- Spec 升级：暂停相关代码修改，等待新 Spec 评审通过。
- 新版本：原代码如已合并则保持，新功能按新 Spec 开发；如未合并则回退或废弃。
