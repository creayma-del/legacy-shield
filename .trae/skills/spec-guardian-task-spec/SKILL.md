# Spec Guardian — Task Spec

本子 Skill 定义任务 Spec 的拆解规则、与阶段 Spec 的关系、评审规则及边缘场景处理。

## 任务 Spec 拆解规则

执行计划评审通过后，必须将执行计划中的每个独立任务拆解为对应的任务 Spec。

### 拆解原则

- **一任务一 Spec**：执行计划中的每个任务（T1, T2, ...）必须对应一个独立的任务 Spec。
- **依赖先行**：被依赖的任务 Spec 应优先编写和评审。
- **可并行评审**：无依赖关系的任务 Spec 可并行评审。
- **不可分割**：一个任务只能对应一个任务 Spec；若任务过大，应先拆分执行计划中的任务。

### 生成步骤

1. 确认执行计划状态为「已通过」。
2. 确认阶段 Spec 状态为「已通过」。
3. 按执行计划任务列表，逐个创建任务 Spec。
4. 任务 Spec 文件头必须引用：
   - 阶段 Spec 路径
   - 设计文档路径
   - 执行计划路径
   - 执行计划中的任务编号和依赖关系
5. 将任务 Spec 初始状态设为「草稿」。

## 任务 Spec 与阶段 Spec 的关系

| 维度 | 阶段 Spec | 任务 Spec |
|---|---|---|
| 范围 | 定义版本总体目标、范围、验收标准 | 定义单个任务的具体实现步骤、测试计划、验收标准 |
| 引用 | 引用需求文档、设计文档、执行计划 | 引用阶段 Spec、设计文档、执行计划中的具体任务 |
| 修改权限 | 阶段 Spec 通过后不可直接修改 | 任务 Spec 通过后不可直接修改；如需变更，按范围变更流程处理 |
| 开发门禁 | 所有任务 Spec 通过前，禁止进入代码开发 | 单个任务 Spec 通过前，该任务禁止进入代码开发 |

## 任务 Spec 评审规则

### 评审方式

- 每个任务 Spec 独立评审。
- 任务 Spec 评审可复用阶段 Spec 的评审专家团。
- 任务 Spec 评审不通过时，仅影响该任务；不影响其他已通过任务 Spec 进入开发。

### 整体开发门禁

- 阶段 Spec 状态为「已通过」。
- 所有任务 Spec 状态均为「已通过」。
- 任一任务 Spec 处于「草稿」或「评审中」→ 整体不可进入代码开发。

### 任务 Spec 变更

- 任务 Spec 评审通过后，如需变更，按 Spec 批准后范围变更处理规则执行。
- 任务 Spec 的变更不得导致阶段 Spec 的总体目标或验收标准发生变化；若会，必须升级阶段 Spec。

## 任务 Spec 边缘场景处理

### 执行计划中途变更

若执行计划在阶段 Spec 已通过后发生变更：

- **新增任务**：按新任务创建任务 Spec，单独评审通过后方可开发。
- **删除任务**：若任务 Spec 尚未进入开发，标记为废弃；若已进入开发，按范围变更流程处理。
- **调整依赖**：重新评估受影响任务 Spec 的依赖关系，必要时重新评审。

### 任务合并或拆分

- **合并**：多个任务合并为一个任务时，保留一个任务 Spec，废弃其他任务 Spec，并重新评审。
- **拆分**：一个任务拆分为多个任务时，原任务 Spec 废弃，为每个新任务创建新任务 Spec 并评审。

### 阶段 Spec 变更后任务 Spec 如何同步

- 阶段 Spec 变更后，所有已创建的任务 Spec 必须重新评估一致性。
- 若阶段 Spec 变更影响某个任务 Spec 的目标或范围，该任务 Spec 必须重新评审。
- 阶段 Spec 变更应尽量避免；如必须，应优先在阶段 Spec 评审阶段完成。

### 任务 Spec 评审不通过的处理

- 单个任务 Spec 评审不通过，仅影响该任务进入开发。
- 其他已通过的任务 Spec 不受影响，可并行开发。
- 阶段 Spec 保持「已通过」状态，但版本整体视为「部分任务待评审」。
