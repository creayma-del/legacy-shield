# Spec Guardian — Review

本子 Skill 定义 spec-guardian 的评审审批角色、准入标准、准出标准、重评规则及评审中发现小问题的快速修订规则。

## 评审审批角色

| 文档 | 审批者 |
|---|---|
| 需求文档 | 用户 / 产品负责人 |
| 需求对齐会议纪要 | 用户 / 产品负责人 + SOLO Coder（双方签字） |
| 需求分解文档 | 项目负责人（或代理角色） |
| 设计文档 | 项目负责人（或代理角色），技术相关需架构负责人 |
| 执行计划 | 项目负责人（或代理角色） |
| 阶段 Spec | Spec 评审专家团 |
| 任务 Spec | Spec 评审专家团（可复用阶段 Spec 评审专家） |
| 验收报告 | 测试验收专家 + 用户 / 产品负责人 |

> 在 IDE / 聊天场景下，「审批」指相关人员明确回复「批准」「通过」「同意」等文字确认。

## 强制评审的文档

以下文档在作为下一阶段输入之前，**必须**完成「草稿 → 评审中 → 已通过」的完整流转：

- 设计文档
- 执行计划
- 阶段 Spec
- 任务 Spec
- PATCH 任务 Spec（或阶段 Spec 中的 PATCH 小节）

需求文档、需求对齐会议纪要、需求分解文档虽不强制走同一状态流转，但需求分解文档必须获项目负责人批准，会议纪要必须双方确认，方可进入下一阶段。

## 评审准入标准

文档只有满足以下条件，才可从「草稿」流转为「评审中」：

- [ ] 文档结构完整，必填字段无空缺。
- [ ] 文件头包含版本、对应上游文档、状态、评审记录占位。
- [ ] 任务 Spec 文件头必须包含任务编号、依赖任务、对应阶段 Spec 引用。
- [ ] 作者已完成自检（如 Spec 评审预审清单）。
- [ ] 所有上游依赖文档状态为「已通过」或「已完成，已归档」（PATCH 除外）。
- [ ] 文档内部引用路径有效。

### 任务 Spec 评审准入补充

- [ ] 对应阶段 Spec 状态为「已通过」。
- [ ] 对应执行计划状态为「已通过」。
- [ ] 依赖任务 Spec 如已存在，应优先评审通过。

## 评审准出标准

评审结论必须为以下之一：

### 通过

- 无 P0 阻塞缺陷。
- 无未解决的 P1 重要问题。
- P2 优化项可遗留，但需明确是否在本次处理。
- 所有评审意见已记录并闭环。

### 不通过

- 存在任一 P0 阻塞缺陷。
- 存在未解决的 P1 重要问题且无法在评审会议中快速修订。
- 文档与上游需求/设计存在重大不一致。

## 评审不通过记录模板

```markdown
## 评审记录

> 评审日期：YYYY-MM-DD
> 评审结论：不通过
> 评审人：{姓名}

### P0 阻塞缺陷

| 编号 | 问题描述 | 位置 | 修订建议 |
|---|---|---|---|
| P0-1 | {描述} | {文件/章节} | {建议} |

### P1 重要问题

| 编号 | 问题描述 | 位置 | 修订建议 |
|---|---|---|---|
| P1-1 | {描述} | {文件/章节} | {建议} |

### P2 优化项

| 编号 | 问题描述 | 位置 | 处理建议 |
|---|---|---|---|
| P2-1 | {描述} | {文件/章节} | {建议} |

### 重评要求

- 作者修订后，将状态改回「评审中」并通知评审人重新评审。
```

## 重评规则

- 评审不通过的文档，作者修订后必须重新提交评审。
- 重评时只需关注未闭环的问题，但若作者修改引入了新的范围变更，需重新全量评审。
- 同一文档最多允许 3 轮正式评审；超过 3 轮未通过，应召开专题会议分析原因。

## 评审中发现小问题的快速修订

若评审中发现的均为 P2 级别优化项，且可在评审会议中当场修订：

- 作者当场修订。
- 记录修订内容。
- 评审结论仍为「通过」，但需注明「含 P2 优化项已现场处理」。

若发现 P0/P1 问题，必须标记为「不通过」，走正式重评流程。
