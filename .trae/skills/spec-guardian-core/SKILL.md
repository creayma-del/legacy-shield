# Spec Guardian — Core

本子 Skill 定义 spec-guardian 的核心流程、状态机、状态核查方法及版本决策树。

## 核心流程

任何 MINOR 及以上版本改动必须遵循：

```
需求澄清 → 需求文档 → 需求对齐会议（双方确认） → 需求分解文档（项目负责人批准） → 设计文档（评审通过） → 执行计划（评审通过） → 阶段 Spec（评审通过） → 任务 Spec 拆解 → 任务 Spec 评审通过 → 代码开发 → 测试验收 → 归档关闭
```

PATCH 级别修复可简化部分流程，但涉及设计或接口变更仍需完整流程。

## 状态机与流转规则

### 文档状态定义

| 状态 | 含义 | 是否可修改 |
|---|---|---|
| 草稿 | 文档初稿，尚未进入评审 | 是 |
| 评审中 | 已提交评审，等待或正在评审 | 是（仅限按评审意见修订） |
| 已通过 | 评审通过，可作为开发依据 | 否，需通过补丁流程或新版本 |
| 已完成，已归档 | 开发验收完成，冻结不再修改 | 否 |

### 允许的状态跃迁

| 从 | 到 | 触发条件 | 执行者 |
|---|---|---|---|
| 草稿 | 评审中 | 文档已完成且作者自检通过 | 作者 |
| 评审中 | 已通过 | 评审专家团判定通过，无阻塞缺陷 | 评审专家 / 项目负责人 |
| 评审中 | 草稿 | 评审不通过，需按意见修订 | 作者（按评审意见） |
| 已通过 | 已完成，已归档 | 代码开发完成且测试验收通过 | 项目负责人 / 验收专家 |
| 任何状态 | — | 已归档后不得回退；如需变更，走新版本或 PATCH 流程 | — |

### 禁止的状态跃迁

- 草稿 → 已通过（必须经评审）。
- 已通过 → 草稿（必须通过新版本或 PATCH 流程）。
- 已完成，已归档 → 任何状态（冻结）。

## 状态核查方法

核查任何文档状态时，优先读取文件头中的状态行：

```markdown
> 状态：草稿
```

### 状态读取规则

- 若文件不存在 → 视为「未创建」。
- 若文件存在但无状态行 → 视为「草稿」，需提醒用户补充状态头。
- 若状态为「已通过」或「已完成，已归档」→ 按对应规则处理。
- 若状态为「评审中」→ 禁止进入下一阶段，需等待评审完成。

### 阶段判断优先级

当用户意图不明确时，按以下优先级判断当前所处阶段：

1. 已归档的 Spec/设计/执行计划存在 → 项目处于维护期，新需求需走新版本流程。
2. 阶段 Spec 状态为「已通过」→ 可进入代码开发阶段。
3. 阶段 Spec 处于「草稿」或「评审中」→ 处于阶段 Spec 编写/评审阶段。
4. 设计文档与执行计划均已通过，但无阶段 Spec → 处于阶段 Spec 编写阶段。
5. 设计文档已通过，执行计划未创建/未通过 → 处于执行计划编写/评审阶段。
6. 设计文档未创建或状态非「已通过」→ 处于设计文档编写/评审阶段。
7. 需求分解文档已完成但未批准 → 处于需求分解阶段。
8. 需求对齐会议纪要已签字但无需求分解文档 → 处于需求分解阶段。
9. 需求文档已完成但无会议纪要 → 处于需求对齐阶段。
10. 只有需求澄清清单 → 处于需求澄清阶段。

## 版本决策树

当用户提出改动时，按以下逻辑判断版本级别：

```
是否破坏现有架构或引入不兼容变更？
  ├─ 是 → MAJOR 版本
  └─ 否 → 是否新增功能、能力增强或监控范围扩展？
      ├─ 是 → MINOR 版本（走完整流程）
      └─ 否 → 是否仅修复 bug、修正文档、性能微调？
          ├─ 是 → PATCH 版本（走 PATCH 流程）
          └─ 否 → 与用户澄清改动性质后再判定
```

### 典型场景归类

| 改动类型 | 版本级别 | 流程 |
|---|---|---|
| 新增框架支持、采集能力 | MINOR | 完整流程 |
| 新增 API / CLI 参数 | MINOR | 完整流程 |
| 架构重构 | MAJOR | 完整流程 |
| Bug 修复 | PATCH | PATCH 流程 |
| 文档错别字修正 | PATCH | PATCH 流程 |
| 性能优化（无接口变更） | PATCH | PATCH 流程 |
| 依赖升级导致接口变化 | MINOR/MAJOR | 视影响判定 |
